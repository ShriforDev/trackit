# @trackit/mobile

Capacitor wrapper around the trackit web UI. Adds background location
tracking and native push notifications on top of the existing PWA.

## What this is

A thin native shell (one `WKWebView` on iOS, one `WebView` on Android)
that loads `https://trackit.itsshri.space`. The web UI is unchanged —
the shell adds platform capabilities the browser can't do:

- **Background location tracking** that survives the app being closed
  (Android foreground service, iOS significant location changes).
- **Push notifications** via FCM/APNs so geofence alerts arrive even
  when the app is backgrounded.
- **Secure credential storage** via Keychain (iOS) and EncryptedSharedPrefs
  (Android), used by the native background worker.
- **Offline location queue** that stores fixes locally when the network
  is down and flushes them when connectivity returns.

The web UI inside the webview is the *same* deployed bundle that
`https://trackit.itsshri.space` serves. We don't bundle a separate copy.
That's how UI updates ship without redistributing the APK.

## Identifiers (locked, don't change)

- **App ID**: `space.itsshri.trackit`
- **App name**: `trackit`
- **Bundle target**: Android first; iOS configured but not actively
  shipping.

Once published, the App ID is permanent — it's the canonical identity
of the app on each store.

## Prerequisites

For Android development:

- **Java 17+** — already installed (`java -version`)
- **Android Studio** — required for first-time platform setup and APK
  building. [Download](https://developer.android.com/studio).
- **Android SDK** — installed via Android Studio's SDK Manager.
  Set `ANDROID_HOME` (or `ANDROID_SDK_ROOT`) to point at it.

For iOS development (Mac only):

- **Xcode 15+** — from the Mac App Store.
- **CocoaPods** — `sudo gem install cocoapods` or via Homebrew.
- An Apple Developer account is **only** required to actually run on
  a real device or distribute. The platform project itself can be
  generated and committed without one.

The `tools/` scripts and `cap` CLI assume `bun` from the repo root.

## First-time setup

From the **repo root** (not from `apps/mobile/`):

```bash
# Install all workspace deps including @trackit/mobile
bun install

# Build the web app once so capacitor.config.ts has a webDir to point at
bun run build:web

# Generate the Android platform project (run once)
cd apps/mobile && bunx cap add android

# Sync the latest web build into the native shell
bunx cap sync android
```

For iOS (must be run on a Mac):

```bash
cd apps/mobile && bunx cap add ios
bunx cap sync ios
```

Both `android/` and `ios/` directories should be committed to git so
CI builds and other contributors don't have to regenerate them.

## Daily commands

All run from `apps/mobile/`:

```bash
# Re-sync the web build into both native shells (run after any web change)
bun run sync

# Open the Android project in Android Studio
bun run open:android

# Open the iOS project in Xcode (Mac only)
bun run open:ios

# Run on a connected Android device or emulator
bun run run:android
```

Or from the repo root:

```bash
bun --filter @trackit/mobile sync
```

## How the build pipeline works

The web → native flow is one-way:

1. `bun run build:web` builds the React app to `apps/web/dist/`.
2. `bunx cap sync` copies that bundle into
   `apps/mobile/android/app/src/main/assets/public/` and into
   `apps/mobile/ios/App/App/public/`.
3. The native build (Gradle for Android, Xcode for iOS) packages the
   shell + the bundled web app into an APK / IPA.

The webview is configured to **prefer the live `https://trackit.itsshri.space`**
when online (see `capacitor.config.ts` → `server.url`). The bundled
copy is a fallback for offline / cold-start scenarios.

This means UI-only changes ship via web deploys without redistributing
the APK. Native changes (new plugins, permission updates) require an
APK rebuild.

## Generated platform folders

`android/` and `ios/` are generated by `cap add` and **are committed
to the repo**. This is the standard Capacitor convention. They contain:

- Native build configuration (Gradle, Xcode project)
- Manifest / Info.plist with permissions
- Splash screen + icon resources
- Capacitor plugin glue

Don't hand-edit these casually — when possible, change `capacitor.config.ts`
and re-run `cap sync`.

Sensitive files inside the platform projects (keystores, signing configs,
local SDK paths) are gitignored — see `.gitignore`.

## Permissions wired in

These are configured in the platform projects so geofence-aware features
work on both OSes:

- **Location** (foreground + background): for live position reporting
  and geofence transitions.
- **Notifications**: for native banners when geofence events fire.
- **Foreground service** (Android): keeps location tracking alive while
  the app is closed.

Each platform's manifest includes a user-facing description of *why*
each permission is needed — required for store submission and good UX.

## Roadmap

This is **Phase 1** — the scaffold. Subsequent phases:

- **Phase 2** — `@capacitor-community/background-geolocation` + offline queue
- **Phase 3** — FCM push token registration + smart routing
- **Phase 4** — Signed APK release pipeline via GitHub Actions
- **Phase 5** — Landing page download CTA + privacy policy + first public APK

iOS distribution is **not part of v1**. The platform project stays
warm in the repo so when an Apple Developer account is added later,
shipping iOS is a plumbing exercise — not a rewrite.
